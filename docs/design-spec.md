# Zilliz Cloud Gloabal Provider and Region 页面 — 设计规格

**状态：** 已批准，可进入规划阶段
**日期：** 2026-07-21
**主要受众：** 企业决策者、技术决策者，以及现有 Zilliz Cloud 开发者

## 目标

创建一个 Zilliz.com 市场营销页面，建立用户对 Zilliz Cloud 全球多云基础设施的信心，并让区域可用性易于核实。该页面不是集群创建流程。

优先级如下：

1. 传达全球覆盖、数据驻留意识和企业级就绪能力。
2. 让访客核实云服务商、区域及部署选项的可用性。
3. 提供清晰的申请新增区域的入口

## 信息架构

### 1. 首屏

- 顶部导航完全复用 zilliz.com 的顶部导航，且纯静态展示即可。
- 聚焦全球基础设施和数据部署位置的主标题。
- 简短的辅助文案，涵盖多云覆盖、邻近部署和数据驻留。
- 以醒目的世界地图作为核心视觉组件。
- 标记显示区域位置，并按云服务商着色：AWS 为橙色、Google Cloud 为红色、Azure 为蓝色。
- 地图下方始终显示图例。
- 选中或悬停标记时，显示位置、云服务商、区域 ID 和部署选项可用性。
- 在桌面端，尽量减少地图水平方向的留白并最大化可见地球范围。在小屏幕上，使用适合触控的地图和紧凑的筛选控件。


### 32. 区域目录

- 筛选条件：云服务商、大区和部署选项。选择某部署选项时，必须保留不支持该选项的匹配区域，让访客能看到其明确的 `Not available` 状态，而不是错误地推断该区域不存在。
- 目录字段：位置、云服务商、区域 ID、Free & Serverless、Dedicated。
- 可用能力使用更小的绿色状态圆点和 `Available` 文案。云服务商圆点必须在视觉上更大，并使用各自的品牌颜色。
- 所有已发布区域都必须可在目录和适用筛选条件中被发现，即使某个部署选项不可用。该能力必须明确显示 `Not available`；不得隐藏区域，也不得将未完整发布的能力标为可用。
- 在移动端，将表格行转换为可读的区域卡片，而不是将可用性隐藏在没有标签的图标后。

### 3. 部署建议

保持简洁，并链接至官方文档以了解详细信息：

- 将集群部署在靠近应用和用户的位置。
- 在可能的情况下，选择与相邻数据和云服务相同的云服务商和区域。
- 优先评估数据驻留和企业安全要求。

### 4. 申请新增区域

- 字段：首选云服务商、首选位置和业务背景。 
- 申请新区必填字段 Provider、Region、Email；按钮为 Submit ；Provider 会联动对应 Region。
- 提交前进行必填项校验。
- 确认信息只说明请求已被接收。
- CRM 或支持工单集成属于独立范围；初始 UI 可使用已达成一致的集成端点，或采用明确标注的非生产提交行为。

### 5. Footer
- Footer 完全复用 zilliz.com 的footer 即可，包括内容，样式且纯静态展示即可。

## 数据模型与唯一事实来源

所有页面数据使用一个结构化、由 CMS 或配置支持的唯一来源。该来源必须提供：

- 云服务商名称和品牌颜色。
- 位置名称、国家/地理区域、区域 ID 和地图坐标。
- Free、Serverless 和 Dedicated 的可用性。
- 发布状态、文档 URL 和最后更新时间戳。

地图、覆盖指标、筛选条件和目录都必须由该来源生成。官方 Zilliz Cloud 文档是部署选项语义和限制条件的事实依据。尤其是，页面必须保留 Free、Serverless 和 Dedicated 之间的差异；不得将 CMEK 等企业能力泛化为适用于所有部署类型。

## 视觉系统

- 与 Zilliz.com 的营销表达保持一致：充足留白、自信的展示型排版、克制的边框，以及一个高密度的视觉焦点（地图）。
- 避免装饰性动效。交互仅限于有意义的标记状态和同步筛选。
- 让云服务商品牌、可用性状态和图例在视觉上彼此区分，并通过文字和颜色共同保证无障碍可读性。

## 状态与错误处理

- 若筛选组合没有匹配的已发布区域，显示明确的空状态，并提供清除筛选条件的操作。
- 缺失能力显示为 `Not available`，同时保留该区域在列表中。
- 如果区域数据源不完整或不可用，保留页面结构并显示可用性更新提示；不得渲染可能产生误导的汇总数量。
- 新增区域模块提交失败时，保留已输入的值，并提供可重试的错误信息。

## 验收标准

- 区域事实、能力标签和文档链接与已批准的官方数据源一致。
- 更改筛选条件后，地图、指标数量和目录会同步且一致地更新。
- 所有已发布区域都仍然可被发现，包括部署选项不可用的区域。
- 桌面端和移动端布局均保留完整标签、可见图例和可读状态。
- 键盘导航、可见焦点、颜色提示的文本替代，以及颜色对比度均满足基础无障碍预期。
- 空结果、缺失数据和申请提交失败均为经过设计且清晰易读的状态。

## 不在范围内

- 集群创建和需要身份验证的控制台工作流。
- 用于区域申请的后端/CRM 实现。
- 对区域路线图或发布日期作出承诺。
